# Pricing Studio — coherence & AI pass

## Problem

Frank's pricing workbench surfaces the same numbers four or five times in
different formats, the driver waterfall mostly shows a single "Customer
mix 100 %" entry, the willingness-to-pay band visually collapses into a
single stack of dots, the per-customer churn under a proposed price is
not visible, and the rationale never explains *why* we still recommend
the price when there are at-risk customers. Two of the three "human"
outputs (PDF, e-mail draft) ship as inert or static — only the memo is
LLM-backed.

The goal of this pass is three-fold:

1. Remove the redundancies so the workbench reads as a single coherent
   story per SKU.
2. Make the three "intelligence" blocks (drivers, WTP band, customer
   fan-out) honest and decision-useful, including a per-option
   stay-vs-churn breakdown and the recovery-vs-loss justification.
3. Wire the briefing LLM end-to-end so the *Decision memo*, *Email to
   Till*, *Branded PDF* and a new inline *AI insights* pane are all
   generated by the same `services/briefing` pipeline rather than
   hard-coded.

## Out of scope

- New trained models, new datasets, new auth surface.
- Action Center / Margin / Forecasting screens.
- Comparable-cluster panel rework (already gated by `meta.blocks.comparable`).

Three components are deleted outright:

  (`RationaleSimple` block) — superseded by the bottom `RationaleMemo`.

  render in `index.tsx` is removed: the hero's `BandStrip` is the only

## Section 2 — Honest drivers, clear WTP, churn breakdown

### 2.1 Drivers

Backend: `scherzinger-platform/backend/services/pricing/recommendation.py`
inside `_compute_drivers`.

After computing the raw contributions, detect the *degenerate* state
(any single contribution > 0.80 after L1 normalisation while all others
are ≤ `1e-3`). When degenerate:

- Set the block status to `degraded` with
  `reason="curve_flat_or_competitor_locked"`.
- Replace the L1-normalised weights with a **heuristic split** built
  from *signals that don't depend on the win-prob curve or competitor
  data*:
  - `cost_trajectory`: weight = clamp(|cost_yoy_pct| / 10, 0.05, 0.4)
    (uses `cost_outlook.commodity_trend`).
  - `floor_protection`: weight =
    clamp(max(0, floor_to_rec_distance) / rec_price, 0.05, 0.3).
  - `cluster_confidence`: weight = clamp(cluster_confidence × 0.4, 0.05, 0.3).
  - `customer_mix`: residual share = 1 − sum(above), clamped to ≥ 0.1.
- Re-normalise so the heuristic split sums to 1.
- Always emit *all five driver rows* (zero-weighted ones still listed
  with `tone = "neutral"` and a short reason — never silently dropped).

The frontend `DriverWaterfall` already accepts the typed shape and a
block status; no FE change beyond rendering a small "heuristic split"
badge when status is `degraded`.

### 2.2 WTP band

Frontend `WtpBandStrip` + hero `BandStrip`:

- Domain now spans the union of `[floor, cost, ceiling, p10, p90, rec]`
  rather than just `[p10, p90]`. Padding 6 % each side stays.
- Minimum spread floor: if `(hi − lo) / lo < 0.05` we pad to ±5 %.
- New markers:
  - `floor` (rose-deep small triangle below the line),
  - `cost` (ink-3 vertical tick),
  - `ceiling` (ink-3 small triangle below the line, same as floor).
- Each marker has a `<title>` tooltip with the value + the source field
  name (e.g. `cost_state.unit_cost`).
- When `wtp.anchored_from_cluster === true` render a chip *inside* the
  strip: `cluster-anchored, n=<n_deals> deals`.

Standalone `<WtpBandStrip>` in `index.tsx` is removed (band lives in
the hero only). The component file stays — only consumer changes.

### 2.3 Customer fan-out + stay-vs-churn summary

Frontend `CustomerFanout.tsx`:

- The per-row chip binds to `row.risk_if_moved` when
  `proposedPriceDecimal` is non-null; otherwise to `row.churn_p`.
- Add a header summary row above the list:
  ```
  At €658.90: 4 customers stay (LTM €420k), 2 at elevated risk (LTM €110k)
              net recovery +€72k/yr · expected loss €18k/yr
  ```
  - Computed entirely from `block.rows[].tone` + `ltm_eur` and
    `block.summary` (new BFF field, see below) when provided.
- Group rows visually into two sections — *Stays* (tones `plain`,
  `warn`) and *At risk* (tone `alert`) — each with its own subheader.
  Group order stays.

Backend `services/studio/customer_fanout.py` (or wherever
`build_customer_fanout` lives — to confirm during implementation):
emit an extra `summary` block per response:

```
summary = {
  "proposed_price": "<decimal-as-string>",
  "stay_count": int,
  "stay_ltm_eur": "<decimal-as-string>",
  "at_risk_count": int,
  "at_risk_ltm_eur": "<decimal-as-string>",
  "net_recovery_eur_yr": "<decimal-as-string>",
  "expected_loss_eur_yr": "<decimal-as-string>",
}
```

The four monetary fields are computed from the customer rows by
multiplying `ltm_eur × Δprice / current_price` (recovery) and
`ltm_eur × risk_if_moved` (expected loss). Existing math primitives
are reused.

### 2.4 "Why are you still recommending this" sentence

The Decision memo (and only the memo — not duplicated elsewhere) gets
an extra rendered line near the top derived from the existing
recommendation block:

```
"Recommending €868.55 even with 2 customers at elevated churn risk:
 expected recovery (+€72k/yr) exceeds expected loss (€18k/yr)."
```

This is a frontend-rendered template that reads
`wb.recommendation.recommended_price`, the new
`wb.customer_fanout.summary` fields, and the active option (if any).
It's hidden when there are no at-risk customers.

## Section 3 — AI wiring (all four)

All four hit `services/briefing`. The existing service already has the
LLM client, persona templates and rate limit. We add three new BFF
endpoints under that service.

### 3.1 `GET /briefing/sku/{aid}` — memo (no change needed, already
LLM)

- Frontend `RationaleMemo`: add a small **regenerate** button to the
  header row (icon `↻`, label `Regenerate`). Calls
  `useBriefing(aid, persona).refetch({forceRegenerate: true})`
  which adds `?regenerate=1` so the service skips its cache.

### 3.2 `POST /briefing/sku/{aid}/email-draft`

- Body: `{ persona: "till" | "frank" | "manuel", lang: "en" | "de",
            proposed_price?: string, summary?: { ... } }`
- Returns: `{ subject: string, body_md: string,
              persona_used: string, model: string, tokens: int }`
- Frontend: new `EmailDraftDrawer` component, opens when "Email to
  Till" in DecisionFooter is clicked. Drawer has editable subject + body
  fields, a "Copy to clipboard" action and an `mailto:` link.
- Persona prompt template under `services/briefing/prompts/email_till.md`
  etc.

### 3.3 `POST /briefing/sku/{aid}/pdf-draft`

- Body: same as email-draft.
- Returns: `{ exec_summary: string, bullets: string[],
              risks: string[], next_steps: string[], persona_used: string }`
- Frontend: the existing "Branded PDF" button (already a popover with
  persona + lang) now POSTs to this endpoint before opening the PDF
  URL. The PDF renderer (`backend/services/pricing/pdf.py`) is
  updated to accept the narrative block and compose a 1-page layout
  with the exec summary at top, bullets/risks/next-steps as three
  columns. Falls back to the legacy hard-coded layout when the
  briefing call fails (graceful degradation).

### 3.4 `GET /briefing/sku/{aid}/insights`

- Returns: `{ gains: Insight[], risks: Insight[], watch: Insight[] }`
  where `Insight = { headline: string, body_md: string }`.
- Frontend: new `AiInsightsPane` rendered between `DriverWaterfall`
  and `WtpBandStrip` in `index.tsx`. Three side-by-side cards (gains
  green-toned / risks rose-toned / watch ink-toned) with a small
  **regenerate** action and a "Powered by briefing service" footer
  chip.

All four endpoints use the same `BriefingClient`, the same auth, the
same rate limit (60/min per user), and the same `meta.blocks.X` status
shape: `live | empty | degraded | locked`. Each endpoint caches its
response for 24h keyed on `(aid, persona, lang)` with a
`force_regenerate` query flag that bypasses the cache.

## Testing

- Unit tests for the heuristic driver split (degenerate detection, weight
  bounds, zero-weight rows still emitted).
- Snapshot tests for `WtpBandStrip` with the three regimes:
  collapsed (p10≈p50≈p90), spread, anchored-from-cluster.
- `CustomerFanout` Vitest: per-row chip switches from `churn_p` to
  `risk_if_moved` when `proposedPriceDecimal` is set; header summary
  renders the right counts/€ totals.
- Pytest for each new briefing endpoint: cache miss → LLM, cache hit
  → no LLM, persona override, lang override.
- One Playwright e2e: pick Cost-floor → STAYS/AT RISK sections render
  → click "Email to Till" → drawer opens with body_md from LLM (LLM
  call mocked at the briefing client boundary).

## Migration / rollout

No DB migration. New BFF endpoints land behind no feature flag — they
return `meta.blocks.X.status` so the frontend handles `locked` /
`empty` / `degraded` gracefully if the LLM is misconfigured.

Frontend changes are additive except the three deletions
(`RecommendationKpiTiles`, hero rationale aside, standalone WTP strip),
which are wrapped under a single commit so the visual change is
atomic.
